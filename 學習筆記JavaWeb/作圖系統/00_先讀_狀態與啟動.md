1. 使用者要求任何會修改圖片、視覺規格、候選、樣板、圖片契約、QA紀錄或作圖系統狀態的功能時，在本次功能第一次寫入前執行交接前置檢查；明確指令與自然語言要求都適用。
2. 只讀查看、分析、規劃或檢查狀態時不取得工作權，也不修改交接資料。
3. 找到同一工作的交接系統時：
   - 本對話已是目前工作者：直接繼續，不重複接手。
   - 狀態允許接手且沒有工作者：依該交接制度自動以最低成本接手；符合零差異返回時，不重讀不必要資料、不寫接手日誌、不增加世代。

4. 自動接手只執行既有交接制度，不得修改、降低、跳過或繞過其條件。
5. 找不到交接系統時，本作圖系統可獨立運作。
6. 交接系統的手動接手指令仍可使用，但不是正常使用作圖功能前的必要操作。
7. 首次建立的只讀分析階段不接手；使用者確認正式建立後，在第一次寫入前才執行本節。

## 啟動判斷

依序執行第一個符合的流程：

1. `系統狀態＝未建立`：使用者要求正式建立時，讀`80_按需/00_首次建立.md`。
2. 只查看或檢查狀態：讀本文件及要求涉及的最少文件，不改狀態，也不取得交接工作權。
3. 建立或比較候選方案、確認樣板：讀`21_候選方案登錄.md`及`80_按需/10_候選方案與樣板確認.md`。
4. 開始單張圖片：讀`01_通用作圖流程.md`、`11_本工作視覺規格.md`及`12_當前圖片與批次.md`。
5. 開始批次：必須`是否允許批次＝是`，並讀`80_按需/40_批次製作.md`。
6. 發生錯誤、規則衝突、重複失敗或成果漂移：停止受影響部分，讀`80_按需/20_異常與失敗處理.md`。
7. 使用者變更長期規格、樣板或驗收方式：讀`80_按需/30_規格更新與影響評估.md`。
8. 使用者提供新版通用作圖系統：讀`80_按需/50_系統更新.md`；不得當成普通視覺規格更新。
9. 外部SOP、品牌指南或其他權威規格新增或更新：讀`13_外部規格與來源.md`及`80_按需/60_外部規格同步.md`。
10. 首次建立、規格定案、樣板確認、批次完成、重要阻塞或準備交接：若有交接系統，讀`80_按需/70_交接系統協作.md`。

## 明確指令

訊息第一個非空白字元為全形`！`或半形`!`，且第一段正好是下列名稱時，視為明確指令。自然語言仍可表達相同意思。

| 指令 | 執行條件 |
|---|---|
| `！首次建立` | 只限未建立 |
| `！開始作圖` | 工作規格已建立，且來源與本張契約可用 |
| `！測試候選` | 已說明候選目的、差異及比較方式 |
| `！確認樣板` | 指定樣板與確認範圍；不得由AI代替使用者主觀確認 |
| `！批次製作` | 已有適用的確認樣板，且允許批次 |
| `！檢查圖片` | 指定成果；預設執行最低充分檢查 |
| `！更新規格` | 指定變更或提供新規則；先做影響評估 |
| `！更新作圖系統` | 已提供同一模組的新版套件；讀系統更新流程 |
| `！同步外部規格` | 已指定外部規格來源，且來源可讀 |
| `！同步交接摘要` | 同一工作存在交接系統；若尚未取得工作權，首次寫入前自動依該制度檢查並接手 |
| `！檢查狀態` | 只讀，不修改成果 |

條件不符時不猜成其他寫入功能；只回報不符條件與一個最短下一步。

## 每次動手前的必要閘門

在修改圖片前，必須能從`12_當前圖片與批次.md`直接回答：

1. 正在處理哪個來源與哪個成果？
2. 使用哪個視覺規格及確認樣板？
3. 本張適用哪些強制規則？
4. 哪些部分可調整？
5. 使用哪個候選方案或正式方案？
6. 完成前如何驗證？

缺少任何一項且會影響成果時，不得靠推測開始製作。

使用者不必先要求AI「建立契約」。若作圖意圖與必要輸入已清楚，AI應自行建立最短充分契約並繼續；只有缺少會實質改變成果且沒有安全預設的資訊時才詢問。

## 完成前的必要閘門

不得只說「已檢查」。交付前必須：

1. 依本張契約逐項核對強制規則。
2. 對每項提供最小必要證據或標示`未驗證`。
3. 區分`內部QA通過`與`使用者已確認`。
4. 發現偏離時修正或停止，不得修改規則讓錯誤成果變成合格。

## 資訊衝突權威

1. 使用者在目前對話中的明確指示。
2. `12_當前圖片與批次.md`中經授權的本次例外。
3. `13_外部規格與來源.md`中標為`權威來源`且目前有效的外部規格原文。
4. `11_本工作視覺規格.md`中的已確認現行規則。
5. `20_樣板與元件登錄.md`中已確認且適用的樣板／元件。
6. `10_本工作做圖說明.md`。
