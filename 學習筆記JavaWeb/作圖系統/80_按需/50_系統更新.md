# 按需：作圖系統更新

> 本流程只更新通用作圖系統，不等同於更新某份工作的視覺規格。未提供新版系統套件時不得執行。

## 1. 更新前

1. 讀新舊`module_manifest.json`，確認模組代號相同。
2. 確認新版相容性、入口文件與現役檔案清單。
3. 將舊檔分為：
   - 系統制度檔：由新版取代。
   - 工作資料檔：`10`、`11`、`12`、`20`、`21`、`30`及內部歷史，按新版格式轉換。
   - 使用者成果與來源：原地保留，不因系統更新重做或搬移。
   - 未知檔案：列入待核對，不混入新版現役制度區。
4. 現有工作若正在寫入圖片，先停止更新；不得同時修改成果及系統。

## 2. 建立新版

1. 在獨立暫存資料夾建立新版，不直接覆蓋現役系統。
2. 使用新版manifest列出的制度檔，不搬入被取代的舊制度檔；檔名不同但控制角色相同也視為被取代。
3. 轉換工作資料，保留工作代號、規格、外部來源登錄、樣板、候選、紀錄及成果引用。
4. 舊版資料夾保持原樣；新版完成前仍由舊版負責。
5. 不把舊制度文件複製進新版`90_內部歷史`；舊版資料夾本身就是完整備查來源。

## 3. 殘留與一致性檢查

更新完成前確認：

- 實際現役檔案與新版manifest完全相符。
- 沒有兩份入口、兩份通用流程或同功能的新舊制度檔。
- 沒有舊狀態名稱、舊指令、舊路徑或已淘汰引用。
- 系統版本在manifest、`00`、`01`至`03`及使用者說明一致。
- 工作代號、視覺規格版本、樣板版本、候選及圖片紀錄沒有被重設。
- 外部規格來源、權威程度、版本與最近同步狀態沒有遺失。
- 候選與正式規則仍分開。
- 技術QA與使用者確認仍分開。
- 使用者成果及來源沒有被重新驗收、重算或搬移。

任何一項失敗時，新版保持暫存，不切換正式名稱。

額外檔案若是工作附件且仍被`10`至`13`引用，可以原樣保留；無法判定或疑似舊制度檔時留在舊版，不搬入新版現役區。

## 4. 切換

1. 完成新版檢查後，將舊版改名為`舊版_已取代_禁止使用`。
2. 將新版切換為正式作圖系統位置。
3. 在新版`30_製作與QA紀錄.md`追加一筆系統更新事件，只記版本、轉換結果及未處理事項，不重寫圖片歷史。
4. 若第二步失敗，立即恢復舊版正式名稱，不留下兩份可同時使用的系統。

## 5. 未來作為外掛時

若由AI工作記憶與交接平台2.x管理外掛，主平台負責安裝、相容性與名稱切換；本流程仍負責本模組內的工作資料轉換及殘留檢查。外掛不得自行修改核心交接制度或其他外掛，但可經主平台授權同步本模組的工作摘要。
