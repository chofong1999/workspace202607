# 按需：交接系統協作

> 本流程讓交接記憶知道作圖工作發生了什麼。它不授權本系統修改交接制度核心，也不取代交接系統自己的接手與交接流程。

## 1. 適用結構

同一專案根目錄可包含：

```text
專案根目錄/
├─ 交接資料夾/
└─ 作圖工作或作圖系統資料夾/
```

資料夾名稱可不同，以各自入口文件及實際路徑識別，不依名稱猜測。

## 2. 寫入前自動交接前置檢查

1. 只查看、分析、提出導入規劃或檢查狀態時，不接手、不修改交接資料。
2. 使用者要求任何會修改圖片、視覺規格、候選、樣板、圖片契約、QA紀錄或作圖系統狀態的功能時，不論使用明確指令或自然語言，都在本次功能第一次寫入前執行本節。
3. 找不到交接系統時，作圖系統可獨立運作。
4. 找到交接系統時，先確認工作身分相符，再依其現行制度處理：
   - 本對話已是目前工作者：直接繼續，不得重複接手。
   - 狀態允許接手且目前沒有工作者：自動使用該制度允許的最低成本方式取得工作權；符合零差異返回時，不重讀不必要文件、不寫接手日誌、不增加世代。
   - 其他員工正在工作、狀態異常、身分不一致或不允許安全接手：停止正式寫入，只回報真正阻塞原因。
5. 自動取得工作權只負責執行既有交接制度；不得修改制度核心、降低條件、略過檢查或繞過其他工作者。
6. 交接系統的手動接手功能仍可使用，但不得成為正常作圖功能前要求使用者額外操作的必要步驟。
7. 首次建立的只讀分析階段不接手；使用者確認正式建立後，在第一次寫入前才自動執行本節。

## 3. 應同步的時機

- 作圖系統首次建立完成。
- 外部權威規格或長期視覺規格實質變更。
- 使用者確認或淘汰重要樣板／候選。
- 批次開始、暫停或完成。
- 發生會影響下一位AI的重要阻塞、異常或待決定事項。
- 使用者要求工作告一段落並交接。

普通單張進度、每輪QA細節及大量技術輸出不逐筆同步。

## 4. 最小交接摘要

交接系統只需保存：

```text
- 作圖系統位置與系統版本
- 視覺規格版本
- 外部權威規格來源ID與版本
- 已確認樣板版本
- 當前圖片／批次與狀態
- 最新重要決定或阻塞
- 下一步
- 詳細資料入口
```

完整視覺規則、候選內容及逐張QA仍由作圖系統保存，避免重複與版本漂移。

## 5. 寫入邊界

- 可以：依交接制度更新該工作的背景、規則摘要、當前工作、使用者待辦及交接日誌。
- 不可以：自行修改交接制度版本、核心流程、指令定義、員工規則、格式規則或版本升級制度；自動接手不得被解讀為修改或放寬交接條件。
- 交接日誌何時追加、世代何時增加、員工狀態如何變更，完全遵守交接系統。
- 作圖系統不得因同步失敗而假裝已交接；在`12`標記`尚待同步`並回報。

## 6. 衝突處理

交接系統負責「誰在工作、是否可接手、整體下一步」；作圖系統負責「圖片如何做、採用何種規格、樣板與QA」。摘要與詳細資料不一致時，停止受影響工作，回到各自權威文件核對後再同步。
