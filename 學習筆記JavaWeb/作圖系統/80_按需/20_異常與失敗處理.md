# 按需：異常與失敗處理

## 1. 先分級

### 可局部自動處理

- 明確的小範圍位置、裁切、檔名或格式錯誤。
- 修正不改變內容、規格、樣板及既有成果。

處理後執行`QA-CHANGE`。

### 需要建立候選或調整方法

- 現行方法連續兩次因相同原因失敗。
- 新內容超出確認樣板。
- 工具限制使原方法不穩定。

停止擴大，建立候選或替代方法；不得默默降低標準。

### 需要使用者決定

- 規則互相衝突且權威順序無法解決。
- 會改變確認樣板、視覺結果、內容、成果位置或驗收標準。
- AI無法客觀判定主觀品質。
- 需要覆蓋、移動或刪除既有成果。

將多個同源問題合併成最少問題，附選項及影響。

### 立即停止

- 來源可能被破壞。
- 操作目標或權限不明。
- 成果可能寫錯專案或覆蓋正式檔。
- 同一錯誤在批次中重複，可能是系統性問題。

## 2. 禁止的錯誤處理

- 不得重複完全相同且已證明失敗的方法。
- 不得修改規則讓錯誤成果變成合格。
- 不得將`達迭代上限`寫成完成。
- 不得只因檔案能開啟就宣稱內容正確。
- 不得把所有異常都轉交使用者；能低風險自動修正的先修正。
- 不得為單張偶發錯誤立即增加大量長期規則。

## 3. 系統性問題

符合任一項時視為候選系統性問題：

- 相同問題出現在兩張以上圖片。
- 同一規則反覆被誤解或無法執行。
- QA使用與製作相同的錯誤假設而產生假通過。
- 確認樣板本身無法覆蓋常見情況。

處理：

1. 暫停受影響批次。
2. 保存最小失敗證據。
3. 判斷問題位於來源、規格、樣板、工具、流程或QA。
4. 提出最小修正。
5. 依`30_規格更新與影響評估.md`判斷舊成果。
6. 重新驗證樣板後才恢復批次。

## 4. 回報格式

```text
發生什麼：
影響哪些成果：
目前是否安全：
已停止什麼：
可自動處理部分：
需要使用者決定：
建議的最小下一步：
```
