# 程式碼截圖標註規範與 SOP

版本：2.4  
狀態：JSP 樣板與 EL 兩種候選方案測試中  
適用範圍：`workspace202607` 內 Java Web 教學用 JSP 截圖，以及 JSP 內嵌的 HTML、CSS、JavaScript、Java scriptlet、JSP Action、JSTL、EL

目前批次只處理 JSP。獨立 `.java` 檔通常是單一 Java 語言，不列入例行標註；只有使用者另行指定時才處理。

## 2. 顏色與語言對照

| 顏色 | 語言／用途 |
|---|---|
| 藍色 | HTML、HTML attribute、HTML 結構外框 |
| 紫色 | CSS |
| 黃色 | JSP 指令、JSP 註解 |
| 橘色 | JSP Action，例如 `<jsp:useBean>` |
| 紅色 | Java scriptlet，例如 `<% ... %>` |
| 青色 | JavaScript，例如 HTML 事件屬性中的 `onclick="return confirm(...)"` |
| 綠色 | JSTL，例如 `<c:forEach>`、`<c:choose>` |
| 粉紅色 | EL，例如 `${product.name}` |

說明框、連接線、群組括號必須使用和被說明項目相同的顏色。

## 3. 核心視覺規則

### 3.1 外框只框有意義的範圍

- 外框右邊界以該區塊最長的實際程式碼為準，再保留約半個至一個字元的空隙。
- 不可為了對齊說明框而把大量空白一起框入。
- 垂直邊界必須放在兩行文字之間，不得穿過字元、底線或行號。
- 多行區塊使用一個群組外框；不要把每一行都切成獨立小框。
- 上下相鄰但分類不同的外框之間，必須保留至少 2 px 可見間隙；不得共用或互相覆蓋水平框線。
- 巢狀框不得與外層框共用邊界；四周至少內縮 6 px。若實際程式碼太長，應擴大外層框，而不是讓兩框疊線。
- 每個外框在繪製前必須記錄：第一個非空白字形的左界、最後一個非空白字形的右界、第一行字形上界、最後一行字形下界，以及產生右界的最長實際程式碼行號。
- 右邊界必須以框內各行「最後一個實際字形像素」的最大 x 值計算，再保留約半個字元格；禁止用固定寬度、說明框位置或目測空白決定。
- 下邊界必須以語意區塊的結束行字形下界計算。單純相鄰但無結構關係的程式不得藉由空白行包成一大框；但同一外層語意結構（例如從 `<div>` 開始到對應 `</div>`、從 `<table>` 到 `</table>`）可包含其內部排版空白與巢狀語言，應優先用一個外層框呈現完整結構，不得因內部空白行機械拆成零散小框。
- 判斷是否可跨空白行的依據是成對開始／結束標籤或明確語意區塊，不是視覺距離。QA 必須記錄外層區塊的開始行、結束行與配對標籤。
- 上、下、左、右四個邊界都要由本張原圖重新量測，不得沿用上一張圖或同頁其他區塊的固定尺寸。

### 3.2 EL 目前保留兩種候選方案

目前尚未決定唯一正式方案。測試圖片必須明確記錄使用下列哪一種候選方案；兩種方案都不得寫成已確認的唯一標準，也不得寫成已淘汰。使用者確認前不得開始批次製作。

#### 候選 A：粉紅字形方案

1. 保留原始字形、字距與抗鋸齒輪廓，只將完整 EL 運算式的原始字形像素改為粉紅色階。
2. 不加底色、不重打文字、不改變字形位置，也不得染到左右相鄰引號、標籤或其他運算式。
3. 每個 EL 必須由原圖分別量測完整 `${...}` 的實際像素界；禁止使用固定字寬公式批量推算。
4. QA 必須檢查完整字形是否全部變為粉紅、是否漏染、是否誤染相鄰字元，並以獨立於製圖座標的證據驗證。

#### 候選 B：保留原字＋粉紅下劃線方案

1. 完全保留原始截圖中的 EL 字形、語法顏色、抗鋸齒、字距及背景。
2. 在完整 EL 運算式正下方畫 2 px 粉紅色水平線。
3. 水平線左右端點各向下延伸 3 px，明確表示 EL 的起點與終點；端點不得向上碰觸字形。
4. 標記線置於該行字形下界與下一行字形上界之間；與字形至少保留 1 px，與下一行也至少保留 1 px。
5. 下劃線不得穿字、壓字、超過 EL 左右界，也不得包含相鄰引號、標籤或其他運算式。
6. QA 必須檢查原字完全未變、下劃線與端點位置、左右相鄰字元及下一行間距。

#### 兩方案共同規則

- 同一群組有多個 EL 時，右側使用粉紅色群組括號，並連到粉紅色說明框。
- 每個 EL 必須從原圖分別量測實際左右界，不可只靠固定公式批量推算。
- 同一張圖只能套用一種候選方案及其對應 QA；禁止用兩套互斥標準驗收同一張圖。
- 重製時若從原圖恢復程式字形，不得切斷既有 HTML、JSTL 或其他分類框線；全圖 QA 必須檢查框線連續性。

### 3.3 巢狀語言的畫法

- HTML 中的 EL：外層維持藍框，EL 依該張測試圖指定的候選方案標記，粉紅括號連到說明。
- JSTL attribute 中的 EL：JSTL 標籤維持綠框，`${...}` 依指定的 EL 候選方案標記。
- JSP Action attribute 中的 EL：JSP Action 維持橘框，`${...}` 依指定的 EL 候選方案標記。
- CSS 位於 `<style>` 中：CSS 本體使用紫框；`<style>` 開關標籤屬 HTML，可在必要時另以藍色說明，但不可製造重疊框線。
- Java scriptlet：整個 `<% ... %>` 使用紅框；其中的 Java 字串內容不再拆成 HTML 或 EL。
- HTML 事件屬性中的 JavaScript：HTML 標籤與 `onclick` attribute 仍屬藍色 HTML；引號內實際執行的 JavaScript（例如 `return confirm(...)`）使用青色標記並另作語意說明。不得把 JavaScript 誤標為 Java scriptlet，兩者執行位置與語言環境不同。

### 3.4 說明框與標籤

- 說明框優先放在程式碼右側空白區。
- 說明框和任何分類外框至少保留 12 px 間距。
- 說明框不可蓋住其他顏色的框、程式碼、行號或 Eclipse 警告圖示。
- 小標籤不可放在下一行程式碼上方；若空間不足，移到右側並使用連接線。
- 連接線只可通過空白區，不可橫穿文字。
- 連接線的優先順序固定為：調整說明框使其可水平直連 → 單一直線斜連 → 有避障需求時才使用折線。
- 水平或斜線能直達時，禁止套用固定的「水平—垂直—水平」折線模板。
- 折線每一個彎點都必須有避開程式碼、框線或其他說明框的明確理由；沒有理由的彎點一律刪除。
- 同一張圖的連接線應以 0～1 個彎點為原則；超過 1 個彎點必須在 QA 紀錄中說明避障原因。
- 說明文字以 2～3 行為原則，先說「這是什麼」，再說「做什麼」。
- 說明框是給學習者看的教學內容，只能解釋程式碼的語言分類、資料來源、執行行為或畫面結果。不得寫入製圖操作、像素量測、遮罩方式、QA 步驟、框線間距或「如何把字染色」等內部工作規則。
- EL 說明必須指出該張圖中的具體表達式讀取或計算什麼，例如「`${fruits}` 取得水果清單供迴圈使用」；不得只寫泛稱「這是 EL」，也不得用 `${...}` 代替實際程式語意。
- 每個已框選且具有獨立教學意義的語言區塊都必須有可對應的說明。若多個同類區塊合併說明，文字必須明確列出涵蓋的行號或變數，不得讓部分框選區塊沒有解釋。

### 3.5 圖例

- 圖例固定放在右下角空白區；若會遮擋程式碼，改放其他空白區。
- 圖例固定採用標註範例的兩欄四列排列，不得改成單列、橫排或只顯示本圖有使用的分類。
- 左欄由上到下固定為：HTML、CSS、JSP 指令／JSP 註解、JSTL。
- 右欄由上到下固定為：JSP Action、Java scriptlet、EL、JavaScript。
- 即使本張圖片沒有使用某一分類，該分類仍必須保留在圖例中，以維持所有圖片一致。
- 左右內距至少 18 px，上下內距至少 20 px。
- 最底下一列文字與底框至少保留 20 px，不得貼邊。
- 每張圖的圖例尺寸、字體、色塊大小一致。

## 4. 幾何與避讓規則

1. 先量出目前 Eclipse 字型的行高、各行基線及實際字形上下界，不假設固定像素值。
2. 同時量出程式碼第一欄的 x 座標及等寬字型的實際字元格寬度；這兩個值必須由原圖像素取得。
3. 單行框的上、下緣預設各保留 1 px；只有四周沒有相鄰框時才可增加到 3 px。
4. 多行框以上述公式套用到第一行上界與最後一行下界。
5. 單行框上下的視覺留白差不得超過 2 px；若差距更大，視為座標錯誤，必須重畫。
6. 不得只用手動猜測的 `y + height` 決定框線；產生前必須記錄第一行與最後一行的字形上下界。
7. 水平框線必須位於字形以外的行距空隙；相鄰分類框至少相隔 2 px，巢狀框與外框至少相隔 6 px，說明框距離其他框至少 12 px。
8. 不得出現以下情況：
   - 框線穿過程式碼。
   - 標籤遮住下一行。
   - 說明框壓住另一種語言的外框。
   - 連接線從文字中間穿過。
   - 相鄰分類框共用或重疊同一條邊界。
   - 巢狀框的邊界與外層框疊在一起。
   - 連接線存在沒有避障用途的多餘彎點。
   - 標籤或括號超出圖片邊界。
9. 如果空間不足，依序採用：縮短說明文字、移動說明框、改用折線、拆成下一張圖。不可用覆蓋程式碼解決。

## 5. 固定製作 SOP

### 步驟 A：準備原始圖片

1. 從本機 Eclipse 開啟真正的原始檔案。
2. 展開編輯區，盡量一次顯示完整且可讀的程式碼。
3. 儲存無標註原圖到 `picture/01_原始截圖/專案名稱/`。
4. 原圖永遠保留；標註不得直接覆蓋原圖。

### 步驟 B：語言分層分析

逐行列出：

- HTML 範圍
- CSS 範圍
- JSP 指令／註解範圍
- JSP Action 範圍
- Java scriptlet 範圍
- JSTL 範圍
- EL 的精確起點與終點

先判斷外層語言，再判斷內嵌語言。例如 `<td>${product.name}</td>` 的外層是 HTML，內層 `${product.name}` 才是 EL。

### 步驟 C：先做標註配置草圖

1. 畫主要結構外框。
2. 標示巢狀語言；EL 必須先指定候選方案，再依該方案處理，不加填滿底色。
3. 規劃群組括號。
4. 最後安排說明框與連接線。
5. 最後才放圖例。

### 步驟 D：依固定順序繪製

繪製順序固定為：

1. 外層 HTML／CSS／Java／JSTL 等結構框。
2. 內嵌 EL 的候選方案標記與必要的細框。
3. 群組括號。
4. 連接線。
5. 說明框與文字。
6. 圖例。

這個順序可避免後畫的粗框壓住程式碼或較精細的 EL 標示。

### 步驟 E：自動檢查與放大目視檢查

每一張候選圖都必須檢查，不得只抽查。先執行可量化的自動檢查，再依固定用途進行三段目視檢查：

- 100%：檢查全圖構圖、可讀性、說明配置、圖例位置及是否有明顯越界或大片多餘空白。
- 200%：檢查結構細節，包括分類框、框線、連接線、標籤、內距、行尾與空白行邊界。
- 400%：只用於 EL 與相鄰字元的像素級細節；依本圖採用的候選方案檢查漏染、誤染、原字變色、下劃線位置、端點及抗鋸齒邊緣。

自動檢查至少包含：

- 每張測試圖必須記錄使用 `粉紅字形` 或 `原字＋粉紅下劃線`，不得缺省。
- 粉紅字形方案：每個 EL 的字形漏染像素數與相鄰字元誤染像素數必須各自為 0；`$`、`{`、`}`及內容必須逐一驗證，不得把多個 EL 合併成一個總數後通過。
- 原字＋粉紅下劃線方案：EL 原始字形變動像素數必須為 0；下劃線不得超過完整 `${...}` 左右界，且不得碰觸字形、下一行或相鄰字元。
- 說明框與非目標分類框的重疊面積必須為 0。
- 說明框與分類框的安全距離不得小於 12 px。
- 框線穿過字形的像素數必須為 0。
- 單行框上下留白差不得超過 2 px。
- 所有標註元件都必須位於畫布範圍內。
- 每個外框的右界不得小於最長行最後字形右界加半個字元格，也不得多出超過一個字元格的純空白。
- 非結構性外框的下界不得越過最後一個目標非空白行後的行距中線；若外層結構框跨過內部空白行，必須能以成對標籤或明確語意範圍證明其完整性。

目視檢查逐項確認：

- [ ] 每個外框是否只包含實際程式碼，沒有大片多餘空白？
- [ ] QA 是否記錄每個框的首末行、最長行、實際字形右界與最後一行字形下界？
- [ ] 外框是否完整包含最長行行尾，同時沒有納入下一個空白行？
- [ ] 最長一行是否完整包含行尾？
- [ ] 所有水平框線是否位於行距，不穿過文字？
- [ ] 單行框的上下視覺留白是否對稱，差距不超過 2 px？
- [ ] 每個 EL 是否只標到 `${...}`？
- [ ] QA 是否明確記錄本圖採用哪一個 EL 候選方案？
- [ ] 粉紅字形方案：完整 `${...}` 是否全部粉紅，且沒有漏染或誤染左右相鄰字元？
- [ ] 粉紅字形方案：放大至 400% 後，抗鋸齒邊緣是否保持原亮度層次且沒有白色殘影？
- [ ] 原字＋粉紅下劃線方案：EL 原始字形與語法顏色是否完全未被修改？
- [ ] 原字＋粉紅下劃線方案：下劃線是否精確對齊 EL 左右界，且未包含相鄰字元？
- [ ] 原字＋粉紅下劃線方案：放大至 400% 後，下劃線與向下端點是否完全沒有碰觸字形或下一行？
- [ ] 從原圖恢復字形後，所有既有分類框線是否仍保持連續？
- [ ] 是否逐個 EL 同時檢查左端 `${`、右端 `}`及左右相鄰字元，而不是只抽查一端？
- [ ] 說明框是否完全沒有接觸其他分類框？
- [ ] 每個重要框選區塊是否都有對應的學習者說明，沒有只畫框不解釋？
- [ ] 說明文字是否只講程式語意，完全沒有製圖、像素、遮罩或 QA 內部規則？
- [ ] 標籤是否遮住下一行？
- [ ] 連接線是否穿過程式碼？
- [ ] 圖例最底列是否保有足夠內距？
- [ ] 所有標註是否都在圖片邊界內？

任何一項不合格都不能交付為樣板或批次成品。每次檢查結果必須寫入 QA 紀錄，不可只在對話中說「已檢查」。

### 步驟 F：固定迭代 QA 循環

每張圖片的自我修正循環上限固定為 5 輪：

1. 由原始截圖產生候選版本 `r01`。
2. 完成步驟 E 的自動檢查及目視檢查。
3. 如果所有項目均通過，將狀態記為 `PASS` 並停止循環。
4. 如果發現任何問題，將狀態記為 `FAIL`，在 QA 紀錄中寫出問題、原因及預定修正方式。
5. 從原始截圖重新產生下一個版本，不可在錯誤成品上層層修補。
6. 依序使用 `r02`、`r03`、`r04`、`r05`，每個版本都重新執行完整檢查。
7. `r05` 仍有任一問題時，狀態記為 `QA_LIMIT_REACHED`，立即停止該圖片；不得因達到上限而視為通過。
8. 達到上限的圖片移至 QA 未通過區，保留問題紀錄並回報使用者，等待修改 SOP、原始素材或製作策略。

版本命名固定採用：

`檔名_sop2-4_r01.png`

每個候選版本必須有同名的 `.qa.json` 紀錄，例如 `檔名_sop2-4_r01.qa.json`。同一輪不得覆蓋舊候選版本。候選版本與 QA 紀錄放入 `picture/90_內部作業資料_不用看/QA歷程/`；只有 `PASS` 版本才能另外複製到給使用者看的樣板或正式成品資料夾。使用者成果資料夾只放可直接查看的圖片，不放 JSON。

### 步驟 G：樣板閘門

1. 每種主要版型先做 2～3 張樣板。
2. 樣板統一放入 `picture/02_標註成品/00_樣板待確認/`；不得在筆記根目錄另建平行的成果資料夾。
3. 樣板未確認前，不批次製作全部圖片。
4. 使用者指出的是通用問題時，先更新本 SOP，再重做所有受影響樣板。

### 步驟 H：批次製作與一致性檢查

1. 樣板確認後，所有圖片使用相同版本 SOP。
2. 每一張圖片都必須執行最多 5 輪的固定迭代 QA，不得以批次抽查取代單張檢查。
3. 每完成 5 張，額外進行一次跨圖片一致性檢查，確認顏色、圖例、字級、內距、線寬及說明語氣一致。
4. 出現一次系統性錯誤即暫停批次，修正 SOP、提高版本號，並重新處理所有受影響圖片。
5. 單張圖片達到 `QA_LIMIT_REACHED` 時停止該圖片；如果同一問題在兩張圖片重複出現，視為系統性問題並停止整批。
6. 所有給使用者看的標註圖片都必須整合在 `picture/02_標註成品/`，不得於筆記根目錄另建平行成果資料夾。
7. 非樣板成品依驗收狀態分流：`01_未驗收`、`02_驗收未通過_待修正`、`03_已驗收成品`。只有使用者明確確認通過的圖片才能移入 `03_已驗收成品`。

## 6. 完成標準（Definition of Done）

一張圖片只有在以下條件全部成立時才算完成：

- 語言分類正確。
- EL 精確標示、QA 記錄候選方案，且只使用該方案的驗收標準。
- 零文字遮擋。
- 零框線穿字。
- 零說明框與分類框重疊。
- 零多餘大片框選空白。
- 圖例位置、順序與內距一致。
- 已在 100% 尺寸完成全圖構圖、可讀性、說明配置、圖例及整體邊界檢查。
- 已在 200% 尺寸完成分類框、框線、連接線、標籤、內距、行尾及空白行邊界檢查。
- 已在 400% 尺寸依本圖候選方案完成 EL、相鄰字元、端點與像素邊緣檢查。
- 自動檢查所有數值均達標。
- QA 紀錄狀態為 `PASS`，且不存在尚未解決的已知問題。

## 7. 版本修訂方式

修改通用規則時：

1. 說明發現的問題。
2. 修改對應規則或檢查項目。
3. 提高版本號。
4. 記錄哪些舊圖片受影響。
5. 重新產生受影響的樣板，確認後再繼續批次。
6. SOP 版本變更後，舊版 `PASS` 圖片必須標記其使用的 SOP 版本；若新規則會影響舊圖，必須重新進入 QA 循環。
